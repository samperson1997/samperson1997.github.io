数据库KV索引

2019-07-12

KV存储引擎的索引根据key是否有序，分为两种：
[1] 如果key无序，则可以采用hash的方法
[2] 如果key有序，则可以采用B+树的方法，或者类B+树的方法。

散列索引 (hash)

Hash 索引只能够用于使用 = 运算符的相等比较 (但是速度更快)。
Hash 索引不能够用于诸如 <, > 等用于查找一个范围值的比较运算符。
依赖于这种单值查找的系统被称为 “键-值存储”；对于这种系统，尽可能地使用 hash 索引。


[1] 优点：查找(hit)，写入的时间复杂度非常低，性能非常高，要比采用B+树的方法快得多
[2] 缺点：hash的方法为了处理碰撞，空间利用率比较低，占用空间较大。在miss的情况下，hash方法的查找效率会急剧下降
[3] 优化器不能够使用 hash 索引来加速 ORDER BY 操作。这种类型的索引不能够用于按照顺序查找下一个条目。
[4] MySql 无法使用 hash 索引估计两个值之间有多少行 (这种情况由范围优化器来决定使用哪个索引)。
[5] 不支持最左匹配原则，查找某行记录必须进行全键匹配。而 B-tree 索引，任何该键的左前缀都可用以查找记录。

B+树索引

B+树索引可以用于使用 =, >, >=, <, <= 或者 BETWEEN 运算符的列比较。
如果 LIKE 的参数是一个没有以通配符起始的常量字符串的话也可以使用这种索引。
[1] 优点：支持的数据量大，压缩效果好，可以更多的索引放在内存中，以保证每次查询小于等于1的读盘次数
[2] 缺点：索引查找(hit和miss相同)、索引创建的性能、数据库的写入性能慢

最左匹配原则

[1] 针对：联合索引


[2] 最左匹配原则遇上范围查询就会停止，剩下的字段都无法使用索引。
例如a = 1 and b = 2 a、b字段都可以使用索引，因为在a值确定的情况下b是相对有序的；
而a > 1 and b = 2，a字段可以匹配上索引，但b值不可以，因为a的值是一个范围。


[3] b = 2 如果建立(a, b)顺序的索引，匹配不到(a, b)索引；
但是如果查询条件是a = 1 and b = 2或者a = 1(又或者是b = 2 and a = 1)就可以，因为优化器会自动调整a, b的顺序。
再比如a = 1 and b = 2 and c > 3 and d = 4如果建立(a,b,c,d)顺序的索引，d是用不到索引的，因为c字段是一个范围查询，它之后的字段会停止匹配。

聚集索引和非聚集索引

[1] 聚集索引就是以主键创建的索引，非聚集索引(也叫做二级索引)就是以非主键创建的索引。
[2] 聚集索引在叶子节点存放的表中的数据，非聚集索引在叶子节点中存放的是主键和索引列。
使用非聚集索引查询数据时，拿到叶子上的主键再去查找相应的数据（回表）。


[3] 非聚集索引在建立的时候也未必是单列的，可以多个列来创建索引。
创建多个单列(非聚集)索引的时候，会生成多个索引树 (所以过多创建索引会占用磁盘空间)。
[4] 在创建多列索引的时候，涉及到一种特殊的索引：覆盖索引，要查询出的列和索引是对应的，不做回表操作。

DDL操作和DML操作

[1] DDL操作主要包含时间序列索引的相关管理操作。
使用CREATE KVINDEX语句可以创建时间序列，
如：IoTDB > CREATE INDEX ON root.laptop.d0.s0 USING kvindex WITH window_length=3, since_time=0;，
创建了路径为laptop.d0.s0的传感器s0的时间序列的kvindex索引，其中索引的子块长度为3，开始时间为0。


使用DROP KVINDEX语句可以删除时间序列，
如：IoTDB > DROP INDEX kvindex ON root.laptop.d0.s0;，
删除了路径为laptop.d0.s0的传感器s0的时间序列的kvindex索引。


[2] DML操作主要包含用户对时序数据索引的查询操作，KvIndex可以用于对时间序列子序列相似度匹配的加速查询。
使用QUERY KVINDEX语句可以查询时间序列相似子序列，
如：IoTDB > SELECT kvindex(root.vehicle.d0.s0, 1, 100, 0.0, 1.0, 0.0) FROM root.vehicle.d0.s0 WHERE time > 0;，
使用kvindex索引查询路径为laptop.d0.s0的传感器s0的时间序列上和路径为laptop.d0.s0的传感器s0的时间区间为[1,100]的时间子序列模式相似的子序列，其中匹配的容错率为0，倍率为1.0，偏移为0.0。
